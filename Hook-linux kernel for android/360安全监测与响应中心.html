0×01 测试程序


Linux系统中执行命令主要依靠execve函数族，可以通过hook execve函数的方法监控命令执行，首先编写两个测试程序帮助寻找hook。


execute.c:


#include <stdio.h>

#include <stdlib.h>

#include <unistd.h>

 

int main(int argc,char*argv[])

{

    char*newargv[]={NULL,"hello","world",NULL};

    char*newenviron[]={NULL};

    if(argc !=2){

        fprintf(stderr,"Usage: %s<file-to-exec>\n", argv[0]);

        exit(EXIT_FAILURE);

    }

    newargv[0]= argv[1];

    execve(argv[1], newargv, newenviron);

    return0;

}


myecho.c:


#include <stdio.h>

 

int main(int argc,char*argv[])

{

    int i;

    for(i =0; i < argc; i++)

        printf("argv[%d]:%s\n", i, argv[i]);

    return0;

}

将myecho程序路径作为参数传递给execute程序，execute程序将会调用execve函数使用指定的参数执行myecho程序，而myecho程序将会打印出参数。


使用strace跟踪程序运行从而获得程序执行的细节

strace–o output.txt ./ execute ./myecho



首先看到在终端下执行命令是通过execve完成的，Google了程序执行时打开的几个文件后，发现/etc/ld.so.preload这个文件比较重要，可以用来劫持系统调用，并且找到了程序调用的libc库的位置，用IDA打开libc库并查看导出表，可以看到execve函数族还有另外几个相同功能的函数，在Hook的时候必须找到一个合适的Hook点避免遗漏。


通过IDA的xrefs功能可以看到execve函数族的其他成员都是调用了execve函数实现功能，因为我是做病毒分析的，觉得用IDA看更直观一些，这里也可以直接去看glibc源码里的函数代码。


确定了Hook点之后，就可以选择Hook方案了，这时有几个选择：

在应用层：


1、在ring3通过/etc/ld.so.preload劫持系统调用

2、二次开发glibc加入监控代码(据说某产品就是这么做监控的)

3、基于调试器思想通过ptrace()主动注入

 

在应用层做Hook的好处是不受内核版本影响，通用性较好，而且技术难度相对较低，但是缺点更明显，因为ring3层的Hook都是针对glibc库做的监控，只要直接陷入0x80中断，就可以绕过glibc库直接调用系统调用，比如近期分析的某挖矿木马：



既然应用层的监控行不通，那就看看内核层的监控手段：


1、API Inline Hook

2、sys_call_table Hook

3、IDT Hook

4、利用LSM(Linux Security Module)


API Inline Hook以及IDT Hook操作难度较大，而且兼容性较差，利用LSM监控API虽然性能最好，但是必须编译进内核才能使用，不可以实时安装卸载，而sys_call_table的Hook相对易于操作，作为防御者也可以直接从” /boot/System.map-`uname -r` ”中直接获取sys_call_table地址，也可以利用LKM(loadable kernel module)技术实现实时安装卸载，所以最后选择在内核层Hook sys_call_table实现监控。
